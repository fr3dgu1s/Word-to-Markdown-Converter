# SQL Security vs OneLake Security — Gap Analysis

*What**users**give**up**when**adopting**OneLake**Security**mode**,**and**available**workarounds**.**Current**gaps**only**—**excludes**planned**or**in**-**development**items**.*

**Core****principle****:**No per-user grants are permitted in any path. All permissions — data access, write, DDL — must flow through roles. Separation of duties is preserved through role design, not user-level assignment.     **✓****Supported****✗****Not****supported****~****Workaround****only**

|  | GRANT T-SQL GRANT per-user or per-role on tables and objects | ✓ Per-user or role T-SQL GRANT on any object, to any user or role | ~ Roles only, split surface T-SQL GRANT on governed tables is blocked. |  | REVOKE Fine-grained removal of a specific permission from a specific user or role | ✓ Per-user, per-permission T-SQL REVOKE on any object, from any user or role | ✗ Does not exist OneLake Security has no REVOKE command — the model is binary: a role either grants access or it does not. Smaller, targeted roles reduce the blast radius of a role removal. | REVOKE as a command does not exist in OneLake Security. Access is binary — GRANT (role membership) or nothing. There is no way to revoke a single permission from a user while keeping others from the same role. Removing a user from a role removes everything that role carries. T-SQL REVOKE on governed tables is also blocked. |

**Notes**

**(****1****)****Strong****DENY****semantics****—**Explicit DENY override is planned for a future release (RB). Until then, customers must redesign access models around additive role semantics only.

**(****2****)****DDL****permission****control****(****CREATE****/****ALTER****/****DROP****)****—**OneLake Security support for DDL permissions is planned for a future release (RB). Until then, DDL governance requires T-SQL role grants or Workspace Roles — always to roles, never to individual users.

**(****3****)****Dynamic****Data****Masking****(****DDM****)****—**DDM support in OneLake Security is planned post RB. Until then, column exclusion (CLS) is the available alternative — an all-or-nothing result rather than a partial mask.

**(****4****)****Views****and****SQL****object****governance****—**The OneLake Catalog is expected to provide visibility into SQL objects including views. This is a potential pathway for OneLake Security to govern all SQL objects (views, stored procedures, functions) under a single model, closing the current split between SQL-managed objects and OneLake-governed tables.
